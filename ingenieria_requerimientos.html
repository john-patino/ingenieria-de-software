¿Qué es un Requerimiento?

Fundamento conceptual de la Ingeniería de Requerimientos

"Un requerimiento es una condición o capacidad que el sistema debe satisfacer para cumplir con las necesidades, objetivos o restricciones del usuario, del cliente o del negocio."
  • Expresan QUÉ debe hacer el sistema, no CÓMO implementarlo técnicamente.
  • Provienen de diversas fuentes: usuarios, clientes, normativas, regulaciones y estándares del sector.
  • Son la base para el diseño, desarrollo, pruebas y validación del software en todas sus fases.
  • Un error en requerimientos puede costar hasta 60 o 100 veces más si se corrige en producción que al inicio del proyecto.
  • Los requerimientos cambian durante el proyecto: deben gestionarse activamente para evitar el caos.
  • En la mayoría de los casos, los clientes no saben explicar con precisión lo que realmente necesitan — por eso se necesitan técnicas especializadas de elicitación.

Elicitación de Requerimientos

El proceso de identificar y capturar las necesidades del cliente

📘 Fuente: Herrera Betancur — Pascual Bravo (2020)
La elicitación de requisitos es el proceso por medio del cual se identifican las necesidades del cliente para el desarrollo de software. La correcta identificación de estos requisitos representa un punto de partida sólido para el desarrollo del sistema.
¿Por qué es crítica la elicitación?
  • Los problemas en esta etapa afectan directamente todas las demás etapas del ciclo de vida del software.
  • El costo de cambio por no identificar adecuadamente los requisitos puede ser 60 a 100 veces superior al costo de hacerlo bien desde el inicio.
  • Requiere comunicación continua con el cliente y los stakeholders usando múltiples técnicas.
  • El analista debe identificar las necesidades reales del cliente y el impacto del desarrollo sobre personas y organización.

Técnicas de Elicitación

Herramientas para extraer información del cliente y los stakeholders

📘 Fuente: Herrera Betancur — Pascual Bravo (2020)
🎤 Entrevistas y Cuestionarios
Conversación directa con clientes y usuarios para identificar sus necesidades, expectativas y restricciones. Es la técnica más común y fundamental.
💡 Lluvia de Ideas (Brainstorm)
Sesión grupal de generación libre de ideas con el equipo y stakeholders. Permite identificar funcionalidades no evidentes y resolver problemas creativamente.
🔍 Análisis de Sistemas Existentes
Revisión del sistema actual (manual o digital) para identificar sus deficiencias y las mejoras que el nuevo sistema debe incorporar.
📐 Prototipos
Creación de modelos visuales del sistema (wireframes, mockups) para que el cliente valide la comprensión de sus necesidades antes del desarrollo.
👁️ Observación
El analista observa directamente cómo trabajan los usuarios actuales. Permite identificar necesidades implícitas que el cliente no sabe verbalizar.
📄 Estudio de Documentación
Análisis de documentos existentes: manuales, formatos, informes y procedimientos del cliente para entender el contexto del sistema.
🧩 Casos de Uso
Modelado de la interacción entre actores y el sistema. Permiten documentar qué hace el sistema desde la perspectiva del usuario (UML).
🗺️ Mapa de Impacto
Herramienta ágil (Gojko Adzic) que organiza los objetivos del proyecto respondiendo: ¿Por qué? ¿Quién? ¿Cómo? ¿Qué?
🎨 Design Thinking
Metodología centrada en el usuario que combina empatía, definición, ideación, prototipado y prueba para entender profundamente las necesidades reales.

Tipos de Requerimientos

Funcionales y No Funcionales — diferencias fundamentales

✔ Requerimientos Funcionales (RF)
Están relacionados con las funcionalidades que tendrá la herramienta de software. También especifican la comunicación que el software puede tener con otros sistemas, e indican explícitamente lo que el software no debe hacer.
  • Expresan qué hace el sistema para cada tipo de usuario.
  • Se derivan de las necesidades del negocio y del usuario.
  • Son verificables mediante casos de prueba funcionales.
  • Forma: "El sistema [verbo] [objeto] [condición]."
EJEMPLOS
RF01El sistema permitirá al usuario registrarse con correo electrónico y contraseña.
RF02El sistema mostrará el historial de transacciones de los últimos 90 días.
RF03El sistema generará reportes en PDF cuando el administrador lo solicite.
RF04El sistema bloqueará la cuenta tras cinco intentos fallidos consecutivos.
★ Requerimientos No Funcionales (RNF)
No hacen referencia a funcionalidades de la herramienta, sino a características o propiedades que debe tener el sistema, como rendimiento, disponibilidad y seguridad.
  • No describen funciones, sino cómo de bien funciona el sistema.
  • Se verifican con métricas, benchmarks y auditorías.
  • Son transversales: aplican a todo el sistema.
  • Su incumplimiento hace el sistema inaceptable aunque funcione.
EJEMPLOS POR CATEGORÍA
RENDIMIENTOEl sistema responderá en ≤ 2 s bajo 1.000 usuarios simultáneos.
SEGURIDADLos datos se almacenarán cifrados con AES-256 en reposo y en tránsito.
DISPONIBILIDADDisponibilidad ≥ 99,9% mensual (máx. 43 min de caída).
USABILIDADUsuario nuevo completa el registro en menos de 3 minutos.
AspectoFuncionales (RF)No Funcionales (RNF)
¿Qué describen?Funciones, acciones y comportamientosRestricciones y atributos de calidad
¿Quién los origina?Usuarios y clientesArquitectos, regulaciones, estándares
¿Cómo se verifican?Casos de prueba (pasa/falla)Métricas, benchmarks, auditorías
Ejemplo"El sistema registrará pedidos.""Responderá en < 2 segundos."
Si faltan...El sistema no hace lo que se necesitaFunciona pero con mala calidad

Documentación de Requerimientos

Enfoques tradicional (IEEE 830) vs. ágil (Historias de Usuario)

📘 Fuente: Herrera Betancur — Pascual Bravo (2020)
⚡ Las metodologías ágiles valoran "el software funcionando sobre la documentación extensiva", lo que no significa no documentar — significa documentar de forma flexible, liviana y útil para el equipo.
🗺️ Mapa de Impacto (Impact Mapping)
Herramienta ágil creada por Gojko Adzic para alinear los objetivos del negocio con las funcionalidades del sistema.

¿POR QUÉ?

Objetivos del proyecto y metas que se tienen al implementarlo.

¿QUIÉN?

Stakeholders: personas que afectan o son afectadas por el proyecto.

¿CÓMO?

Impacto que tienen los stakeholders y cómo el sistema puede ayudarlos.

¿QUÉ?

Funcionalidades a alto nivel que se deben desarrollar para cumplir objetivos.

📋 Historias de Usuario
La técnica más utilizada en metodologías ágiles para documentar los requerimientos del sistema. En Scrum las escribe el Product Owner; en XP las escribe el usuario.
PLANTILLA ESTÁNDAR
Como [tipo de usuario], quiero [funcionalidad] para [beneficio / valor de negocio].
EJEMPLO RF — Historia de Usuario
Como recepcionista, quiero registrar nuevos pacientes ingresando nombre, cédula, teléfono y correo, para tener sus datos disponibles al asignar turnos y citas médicas.
EJEMPLO RF — Historia de Usuario
Como médico, quiero llamar al siguiente paciente con un solo clic desde mi consultorio, para agilizar la atención sin depender del personal de recepción.
CRITERIOS DE ACEPTACIÓN (RNF incorporado)
Historia: El sistema mostrará el turno actual en pantallas de sala de espera.

Criterio 1: La pantalla debe actualizarse en menos de 3 segundos tras llamar al paciente.
Criterio 2: El texto debe ser visible a 5 metros de distancia (fuente ≥ 48pt).
🔄 Flujo de documentación ágil (Scrum)
PASO 1
Mapa de Impacto
¿Por qué? ¿Quién? ¿Cómo? ¿Qué?
PASO 2
User Story Mapping
Priorizar acciones y funcionalidades
PASO 3
Product Backlog
Lista priorizada de todas las HU
PASO 4
Sprint Backlog
HU seleccionadas para cada Sprint
Herramientas recomendadas para gestionar el backlog: Trello · Miro · Jira

Especificación de Requerimientos

Buenas prácticas y estándar IEEE 830

📋 Referencia: IEEE Std 830 — Especificación de Requisitos de Software

📄 Estructura del documento IEEE 830 (SRS — Software Requirements Specification)

1. IntroducciónPropósito, alcance, definiciones y referencias del documento.
2. Descripción GeneralPerspectiva, funciones principales, usuarios, restricciones y suposiciones.
3. Requerimientos EspecíficosRF detallados, RNF, interfaces externas y restricciones de diseño.
4. ApéndicesInformación adicional, glosario, modelos y diagramas de apoyo.
Características de un Buen Requerimiento
C
Claro
Sin ambigüedad: un solo significado posible para todos los involucrados.
Co
Completo
Incluye toda la información necesaria para diseñar e implementar la solución.
V
Verificable
Se puede probar con criterios objetivos. Evitar: "rápido", "fácil" sin cifras.
Cs
Consistente
No contradice ni entra en conflicto con ningún otro requerimiento.
P
Priorizado
Tiene asignado un nivel de importancia acordado con el cliente.
T
Trazable
Rastreable desde su origen hasta su implementación y prueba.
✔ Funcionales bien redactados
RF01El sistema permitirá al usuario registrarse con correo y contraseña, validando el formato antes de confirmar.
RF02El módulo generará archivos PDF con datos filtrados por el rango de fechas seleccionado por el usuario.
RF03El sistema enviará correo de confirmación en máximo 2 minutos tras completar una transacción exitosa.
RF04El administrador podrá activar o desactivar cuentas desde el panel de administración.
RF05El sistema calculará el total de la factura con los impuestos vigentes según normativa tributaria colombiana.
★ No Funcionales bien redactados
RNF01El sistema responderá a cualquier solicitud en máximo 2 segundos bajo 500 usuarios concurrentes.
RNF02La información se almacenará cifrada con AES-256; las contraseñas como hash bcrypt (costo ≥ 12).
RNF03La interfaz será responsiva y compatible con Chrome, Firefox y Edge en sus últimas dos versiones.
RNF04Disponibilidad mínima del 99,5% mensual (menos de 3,6 horas de inactividad total).
RNF05El código alcanzará al menos 75% de cobertura con pruebas unitarias automatizadas.

✅ Recomendaciones finales al iniciar la elicitación (Herrera Betancur, 2020)

Tener comunicación directa y continua con el cliente y todos los stakeholders del proyecto.
Escuchar primero la descripción completa del problema — esto da visión global antes de entrar en detalles.
Tomar nota y apoyarse en elementos escritos para no olvidar detalles importantes de las conversaciones.
Usar Wireframes para aclarar requerimientos cuando el cliente no puede describirlos verbalmente.
Tener claro y por escrito el alcance del sistema y las condiciones de entrega antes de comenzar el desarrollo.
Tener en cuenta la infraestructura técnica sobre la cual será implementado el sistema desde el inicio.

Ejercicio Práctico en Clase

Actividad grupal de 40–50 minutos para identificar, clasificar y redactar correctamente los requerimientos de un sistema real.

⏱ 40–50 min 👥 Equipos de 3–4 📋 3 Actividades Sistema de Gestión de Turnos Médicos
📄

Estudio de Caso

Lean con atención antes de comenzar

Sistema de Gestión de Turnos y Citas Médicas — Clínica "Salud Total"

Ciudad intermedia de Colombia · ~350 pacientes diarios · Consulta externa, urgencias, laboratorio

Contexto: La Clínica "Salud Total" opera actualmente de forma manual: los pacientes llegan al mostrador, solicitan turno en papel y esperan ser llamados por nombre en voz alta. Esto genera largas filas, confusión entre usuarios, errores en la asignación de turnos y pérdidas de información.

Situación: La directora administrativa, Dra. Martina Vargas, ha contratado un equipo de desarrollo para construir un Sistema de Gestión de Turnos y Citas Médicas. En la reunión inicial (proceso de elicitación) compartió las siguientes necesidades:

NECESIDADES EXPRESADAS EN LA REUNIÓN DE ELICITACIÓN

1
Los pacientes deben poder sacar turno de forma presencial (taquete físico) o desde la aplicación móvil de la clínica.
2
El sistema debe mostrar en pantallas de sala de espera el número de turno actual y el nombre del paciente.
3
Los médicos y enfermeras deben poder llamar al siguiente paciente desde su consultorio con un solo clic.
4
El sistema debe permitir agendar citas médicas con anticipación (hasta 30 días), asignando médico, especialidad, fecha y hora.
5
La recepcionista debe poder registrar nuevos pacientes con sus datos básicos: nombre, cédula, fecha de nacimiento, teléfono y correo.
6
El administrador debe poder generar reportes de atención diaria, semanal y mensual en formato Excel o PDF.
7
El sistema no debe caerse en horas pico (entre 7:00 a.m. y 11:00 a.m. se atienden hasta 120 pacientes por hora).
8
La aplicación debe ser fácil de usar para personas mayores de 60 años, que representan el 40% de los usuarios.
9
Toda la información de los pacientes debe estar protegida conforme a la Ley 1581 de 2012 (habeas data en Colombia).
10
El sistema debe funcionar en tablets, computadores de escritorio y teléfonos inteligentes.
11
El personal administrativo no debe necesitar más de 2 horas de capacitación para usar el sistema.
12
El sistema debe registrar un historial de citas y turnos por paciente para consulta futura por parte del médico.
ActorRol en el sistema
PacienteSolicita turnos y citas; consulta su estado de atención desde la app o en sala de espera.
RecepcionistaRegistra pacientes, gestiona turnos presenciales y apoya el agendamiento de citas.
Médico / EnfermeraLlama al siguiente paciente y consulta el historial de citas desde el consultorio.
AdministradorGenera reportes, configura el sistema y gestiona perfiles de usuario.

Actividades

Completen en orden. Cada actividad construye sobre la anterior.

1
Identificación de Requerimientos
⏱ 15 min

Instrucción: Lean el caso. Identifiquen al menos 10 requerimientos del sistema a partir de las necesidades expresadas. Escríbanlos en la tabla de entregable.

Recuerden: un requerimiento expresa una capacidad o restricción concreta del sistema, no una descripción del proceso actual.

💡 Técnica: Apliquen lo aprendido sobre elicitación — busquen verbos de acción (registrar, mostrar, generar, proteger, notificar, calcular…) en el texto. Cada verbo suele corresponder a un requerimiento.
2
Clasificación con Justificación
⏱ 10 min

Instrucción: Clasifiquen cada requerimiento como RF o RNF. En la columna "Justificación" expliquen por qué lo clasificaron así.

ES RF si...
Describe una acción, función o servicio que el sistema realiza para un actor.
ES RNF si...
Describe una restricción de calidad: rendimiento, seguridad, usabilidad, disponibilidad, portabilidad…
3
Reescritura con Buenas Prácticas
⏱ 15 min

Instrucción: Los siguientes requerimientos están mal redactados. Apliquen las características IEEE (claro, completo, verificable, consistente) y reescríbanlos correctamente.

N°TipoRequerimiento MAL redactadoProblema
1RF"El sistema debe ser rápido al mostrar los turnos."Ambiguo: "rápido" no es medible.
2RF"Los usuarios podrán hacer cosas en el sistema."Incompleto y vago: no especifica qué acciones ni qué usuarios.
3RNF"El sistema debe ser seguro y proteger los datos."Sin estándar, sin tipo de dato, sin mecanismo.
4RNF"El sistema no debe fallar."No verificable. Sin métrica de disponibilidad.

Tips para el Estudiante

Conceptos clave con definiciones de los autores — guarda esto como referencia rápida

📌
Tip 01 · Concepto fundamental
¿Qué es realmente un requerimiento?

La correcta identificación de los requisitos del sistema representa un punto de partida sólido para el desarrollo. Identificar adecuadamente las necesidades del cliente, las personas involucradas en el proceso y el impacto que puede tener el desarrollo en estas personas y en la compañía, traza la ruta para la implementación de las metodologías.

Herrera Betancur, S. (2020) — I.U. Pascual Bravo
Un requerimiento no es lo que el desarrollador cree que el cliente necesita. Es lo que el cliente realmente necesita, expresado de forma clara, verificable y acordada entre ambas partes.
✔ HAZLO ASÍ"El sistema permitirá registrar pacientes con nombre, cédula y correo."
✘ EVITA ESTO"El sistema debe tener un formulario bonito para los datos del paciente."
🎤
Tip 02 · Elicitación
El cliente no siempre sabe lo que necesita

Identificar los requisitos de un sistema requiere de comunicación continua con el cliente y los stakeholders. Las personas encargadas de realizar estos procedimientos deben hacer uso de diferentes técnicas para identificar correctamente las necesidades del cliente y el impacto que el desarrollo de la herramienta puede generar sobre las personas y la compañía en general, ya que en la mayoría de los casos los clientes no saben explicar lo que realmente necesitan.

Herrera Betancur, S. (2020) — I.U. Pascual Bravo
Por eso no basta con hacer una sola entrevista. Combina técnicas: entrevistas, observación, prototipos y talleres. Cada técnica revela un ángulo diferente de la necesidad real.
⚠️
Tip 03 · Impacto económico
Un error temprano se paga muy caro después

El costo del cambio generado por no identificar adecuadamente los requisitos del sistema puede ser 60 o 100 veces superior a si se hubiera realizado el cambio al principio del proyecto.

Herrera Betancur, S. (2020) — I.U. Pascual Bravo
Imagina que construyes 10 pisos de un edificio sobre un plano equivocado. Corregir la base cuando ya está terminado el edificio es devastadoramente costoso. Los requerimientos son los planos del software.
⚖️
Tip 04 · Clasificación
La diferencia entre RF y RNF en una frase

Los requerimientos funcionales están relacionados con las funcionalidades que tendrá la herramienta de software. Los requisitos no funcionales no hacen referencia a funcionalidades de la herramienta, sino a características o propiedades que debe tener el sistema, como por ejemplo rendimiento, disponibilidad, seguridad.

Herrera Betancur, S. (2020) — I.U. Pascual Bravo
Truco rápido: si puedes imaginar un botón o pantalla que lo ejecuta → es RF. Si es una condición de calidad que aplica a todo el sistema sin ser una función visible → es RNF.
RF — ¿QUÉ hace?"El sistema mostrará el turno en pantalla." — Hay una pantalla que hace algo.
RNF — ¿CON QUÉ CALIDAD?"La pantalla se actualizará en < 3 s." — Es una restricción de rendimiento.
🔄
Tip 05 · Metodologías ágiles
Documentar liviano no significa no documentar

Las metodologías ágiles han impactado fuertemente en la forma como se realiza la elicitación de requisitos y su respectiva documentación. A diferencia de las metodologías tradicionales, la documentación es más liviana, ya que según el manifiesto ágil, se valora más los individuos e interacciones sobre los procesos y herramientas, el software funcionando sobre la documentación extensiva.

Herrera Betancur, S. (2020) — I.U. Pascual Bravo
En Scrum, los requerimientos viven en el Product Backlog como Historias de Usuario. No desaparecen — cambian de forma. Son más cortos, más flexibles y se refinan sprint a sprint con el cliente.
✔ ÁGIL: Historia de Usuario"Como médico, quiero llamar al siguiente paciente con un clic, para agilizar la consulta."
TRADICIONAL: IEEE 830RF-014: El sistema proveerá al médico un botón de "Llamar siguiente" en la vista de consultorio, que actualice el turno en curso...
🎯
Tip 06 · Calidad de redacción
Si no puedes medirlo, no puedes verificarlo

Una especificación de requisitos de software debería ser correcta, no ambigua, completa, consistente, clasificada por importancia o estabilidad, verificable, modificable y rastreable.

IEEE Std 830-1998 — Software Requirements Specification
La verificabilidad es el criterio más fácil de evaluar y el más frecuentemente violado. Antes de escribir un requerimiento, pregúntate: "¿Cómo pruebo que el sistema cumple esto?" Si no tienes respuesta clara, reescríbelo.
✔ VERIFICABLE"El sistema responderá en ≤ 2 segundos bajo 500 usuarios simultáneos." — Puedo medirlo con una prueba de carga.
✘ NO VERIFICABLE"El sistema debe ser rápido y fácil de usar." — ¿Qué significa rápido? ¿Fácil para quién?
📝
Tip 07 · Para el proyecto
Checklist antes de entregar tus requerimientos

Tener una comunicación directa con el cliente y los stakeholders. Escuchar la descripción del problema por parte del cliente, esto dará una visión global de lo que se desea construir. Tomar nota y apoyarse de elementos escritos, para no olvidar detalles importantes. En caso de necesitar aclarar algunos requerimientos, se pueden utilizar Wireframes.

Herrera Betancur, S. (2020) — I.U. Pascual Bravo
¿Cada requerimiento tiene un solo significado posible? → Claridad ✔
¿Puedes diseñar una prueba para verificarlo? → Verificabilidad ✔
¿Contradice algún otro requerimiento? → Consistencia ✔
¿Sabes de dónde viene y a qué caso de uso corresponde? → Trazabilidad ✔
¿Tienes claro el alcance y las condiciones de entrega con el cliente? → Alcance definido ✔
